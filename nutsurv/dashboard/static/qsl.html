Questionnaire Specification Language (QSL)

Specification

A questionnaire can be defined using a simple format familiar to anyone who has ever used any outliner (or lists with bullet points).  To see how it would look like in practise please see section Example.

Basic data types

  • string
  • optional: multi-line (do we want it?)
  • integer
  • real - i.e. real numbers
  • date
  • time
  • datetime
  • gps
  • enumeration

Predefined enumerations

  • yes/no
  • enumeration
  • Y
  • N
  • gender
  • enumeration
  • M
  • F

Other reserved words / keywords

  • answer - denotes a structure describing the allowed values and data type of an answer; it can be replaced by the data type identifier in case of most pre-defined data types (but not when defining enumeration)
  • dependencies - this question should only be asked if answers to all the questions contained within it fulfil the constraints listed (see Structure or Example)
  • required - a shortcut for a constraint indicating that answering this question is not optional
  • skip - works analogously to dependencies but has an opposite effect (i.e. specifies conditions for this question to be skipped)

Structure

The structure would be defined by indentation (using tabs or an arbitrary number of spaces).

Please replace the bullet symbol with a dash (-) or (*) (which one would be better received by the user if we decide to implement this?):

  • questionnaire
  • id
  • title
  • description
  • questions
  • question id
  • label / text (i.e. what the person being interviewed is asked)
  • description / comment - additional text describing this question (for internal use of the questionnaire designers, not shown)
  • required - true or false (false means the question is optional - i.e. answer not required)
  • answer
  • type - it specifies acceptable values for an answer to this question (e.g. integer, real, string, multiline(string), date, time, datetime, gender, yes/no, true/false, enumeration)
  • constraints the answer must fulfil (e.g. the value must not be empty)
  • another constraint
  • dependencies
  • some question id
  • some value constraint (e.g. value > 10; or the value is empty (i.e. question not answered))
  • skip - skip this question if the following constraints are fulfilled
  • some question id
  • some value constraint

Notes

  • The sequence would be the same as the order in which questions are defined. Alternatively we could use numerical question IDs to determine the position of a question in the sequence or (if we want to allow for alphanumerical IDs to increase readability) we could add a separate attribute storing sequence priorities (preferably unique) as numerical values.
  • Constraints can be written in a selected programming language (e.g. Python or JS) in case the questionnaire is to be automatically processed (as in our scenario) or in a pseudocode or natural language for further processing by an engineer.
  • Attribute "required" is redundant as it may be easily modelled with constraints (e.g. not empty) but may be worth including if it would make the questionnaire definition process more intuitive.
  • To make it even easier, we could use the question text / label in place of the question id (if their uniqueness can be assumed).
  • We could add any/all to give the user the ability to build composite constraints if necessary but it would increase complexity for both the user and us (do we need such a feature at this stage?).
  • For all predefined data types (as of this writing all but the enumeration) we could allow for simplified notation
  • e.g.:
  • Do you agree?
  • yes/no
  • required
  • instead of:
  • Do you agree?
  • answer
  • yes/no
  • required
